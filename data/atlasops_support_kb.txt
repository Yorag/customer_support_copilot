AtlasOps Support Knowledge Base

Application background:
AtlasOps is a B2B SaaS platform for workspace administration, identity access control, workflow automation, audit reporting, and partner integrations. Customers use AtlasOps to manage employee access, automate approvals, configure webhooks, and export compliance records.

Audience:
This knowledge base is written for support agents who need to answer product questions, explain policy boundaries, collect troubleshooting details, and decide when a case must be escalated to a human reviewer.

================================================================================
Document 1: Product Overview
================================================================================

AtlasOps provides the following core capabilities:
- Workspace administration for user, group, and role management
- SAML single sign-on for supported plans
- SCIM 2.0 provisioning for enterprise customers
- Audit logs and audit export APIs
- Approval workflows for access requests
- CSV user import for initial onboarding
- Webhook notifications for workflow and user lifecycle events
- REST API for workspace automation

AtlasOps is a web application. Customers do not deploy AtlasOps inside their own infrastructure unless they have a custom enterprise agreement. Standard plans are cloud-hosted only.

Supported admin surfaces:
- Web admin console
- REST API
- Webhooks

AtlasOps does not currently support:
- On-premise deployment for self-serve plans
- More than one active identity provider per workspace
- Custom SCIM bridge failover routing by region
- Audit export retention override on Starter or Pro
- Verbal refund approval by frontline support

Document 2: Plans and Packaging
================================================================================

Starter plan:
- Up to 50 active users
- Email and password login only
- No SSO
- No SCIM

- Up to 500 active users

Enterprise plan:

- Plan upgrades take effect immediately
- Plan downgrades take effect at the next renewal date

Document 3: SSO and Identity Management
================================================================================

SSO availability:
- Pro supports SAML SSO
- Enterprise supports SAML SSO
- Starter does not support SSO

Where to configure SSO:
- Go to Settings
- Open Security
- Open Single Sign-On
- Choose the identity provider type
- Enter the SAML sign-in URL
- Enter the X.509 certificate
- Enter the issuer ID
- Save the configuration
- Run the built-in test connection before enforcing SSO

Important SSO behavior:
- AtlasOps allows one active identity provider per workspace
- AtlasOps supports SP-initiated and IdP-initiated SAML login
- Support can explain setup steps but cannot directly modify a customer's SSO configuration
- Enforcing SSO should only be done after at least one workspace admin verifies access

Common SSO login failure causes:
- Incorrect ACS URL
- Expired signing certificate
- Issuer mismatch
- Wrong NameID format
- User not assigned in the identity provider
- Clock skew between the IdP and AtlasOps

SSO troubleshooting checklist:
- Ask which identity provider is in use
- Ask whether the failure affects one user or all users
- Ask for the exact error message
- Ask for the approximate timestamp of the failed login
- Ask whether the customer changed the certificate or issuer recently
- Ask whether the customer tested both SP-initiated and IdP-initiated login

SCIM provisioning:
- SCIM 2.0 provisioning is available only on Enterprise
- AtlasOps provides one standard SCIM endpoint per workspace
- AtlasOps does not support a custom SCIM bridge for regional failover
- AtlasOps does not support multiple active SCIM providers in the same workspace

Document 4: API, Webhooks, and Integrations
================================================================================

API availability:
- Pro and Enterprise plans include API access
- Starter does not include API access

API rate limits:
- Pro: 120 requests per minute per workspace
- Enterprise: 300 requests per minute per workspace
- Burst protection may temporarily reject traffic above the documented per-minute limit
- Rate-limited requests return HTTP 429
- Customers should use retry with exponential backoff

API pagination:
- Most list endpoints return up to 100 records per page
- Cursor-based pagination is recommended for large exports

Webhook behavior:
- Webhooks are available on Pro and Enterprise
- AtlasOps expects the receiving endpoint to return a successful response within 10 seconds
- If the endpoint does not return success within 10 seconds, the delivery attempt is treated as failed
- Failed deliveries are retried after 1 minute, 5 minutes, and 30 minutes
- AtlasOps signs webhook requests with an HMAC signature header

Webhook troubleshooting checklist:
- Ask for the endpoint URL or integration name
- Ask for the approximate failure timestamp
- Ask whether the endpoint returned a non-2xx response
- Ask for any request ID or delivery log visible in the AtlasOps admin console
- Ask whether the endpoint usually responds within 10 seconds

Unsupported integration behavior:
- AtlasOps does not offer guaranteed custom webhook replay windows beyond the default retry schedule
- AtlasOps does not support customer-defined webhook delivery regions on Pro

Document 5: Imports and Data Operations
================================================================================

CSV import:
- CSV import is available on all plans
- Required columns are email, first_name, and last_name
- Optional columns include department, role, manager_email, and start_date
- The maximum CSV import size is 20,000 rows per job

Common import failures:
- Missing required columns
- Duplicate email addresses in the file
- Invalid UTF-8 encoding
- Date values in unsupported formats
- Role values that do not match workspace role configuration

Import troubleshooting checklist:
- Ask for the import job ID if available
- Ask for the approximate time the job failed
- Ask whether the file includes duplicate emails
- Ask for a small sanitized CSV sample
- Ask whether the customer recently changed role mappings

Data export:
- Pro includes on-screen audit logs
- Enterprise includes the audit export API
- Starter does not provide export APIs

Document 6: Audit, Retention, and Compliance
================================================================================

Audit retention:
- Starter retains audit logs for 30 days
- Pro retains audit logs for 365 days
- Enterprise retains audit logs for 365 days by default

Audit export retention override:
- A retention override is not generally available on Starter or Pro
- Enterprise customers may request longer retention only if they purchase the Compliance Archive add-on
- Support should not promise a retention exception unless an approved enterprise add-on is already active

Compliance Archive add-on:
- Available only to Enterprise customers
- Extends audit retention to 7 years
- Requires commercial approval and provisioning by internal operations

Support scope:
- Support can confirm whether a workspace has the Compliance Archive add-on
- Support cannot enable the add-on without commercial approval

Document 7: Billing, Refunds, and SLA Boundaries
================================================================================

Billing policy:
- Support can explain invoice timing, renewal dates, and plan differences
- Support cannot directly approve refunds, credits, or contract amendments
- Duplicate charge reports must be reviewed by finance operations
- Frontline support should collect invoice IDs, charge dates, workspace ID, and a brief description of the issue

Refund policy:
- If a duplicate charge is confirmed by finance, the refund is typically processed within 5 to 10 business days
- Support must not promise a refund before finance confirms the duplicate charge
- Self-serve cancellations stop future renewals but do not automatically trigger prorated refunds

SLA and compensation:
- SLA commitments apply only to Enterprise customers with a signed agreement
- Starter and Pro do not include SLA credits
- Requests for compensation, penalty payments, legal review, or contract exceptions must be escalated to a human team
- Support may acknowledge the request and explain that a specialist will review it

High-risk commercial scenarios that require escalation:
- Refund request for duplicate billing
- Compensation request tied to outage or missed SLA
- Charge dispute with legal or contractual language
- Request for exception to approved billing policy

Document 8: Customer Communication Guidelines
================================================================================

Support agents should:
- Answer factual product questions using the documented feature set
- Be explicit when a feature is unsupported or unavailable on the customer's plan
- Ask focused clarifying questions when troubleshooting details are missing
- Acknowledge customer frustration without being defensive
- Avoid inventing features, undocumented integrations, or roadmap commitments

Support agents should not:
- Promise refunds, credits, or compensation
- Promise custom feature delivery dates
- Claim an issue is fixed without evidence
- State that an undocumented feature is supported

For feedback and complaints:
- Thank the customer for sharing the feedback
- Briefly acknowledge the pain point
- Avoid promising roadmap changes
- If appropriate, offer to log the feedback for product review

Document 9: Troubleshooting Playbooks
================================================================================

Case type: SSO login keeps failing
Recommended next questions:
- What identity provider are you using
- What exact error message do users see
- Does the issue affect all users or a subset
- When did the problem start
- Did you rotate the certificate or change issuer settings recently
- Are you testing SP-initiated login, IdP-initiated login, or both

Case type: User sync or provisioning is broken
Recommended next questions:
- Are you on Enterprise with SCIM enabled
- Which identity provider handles provisioning
- What was the last successful sync time
- Did the SCIM token change recently
- What error appears in the IdP or AtlasOps provisioning logs

Case type: Webhook timeout in production
Recommended next questions:
- Which webhook subscription is failing
- What endpoint receives the webhook
- What response code did the endpoint return
- Does the endpoint usually respond in under 10 seconds
- Do you have a request ID or delivery timestamp

Case type: Import job failed
Recommended next questions:
- What is the import job ID
- Roughly how many rows were in the file
- What step failed, validation or processing
- Can you share a sanitized sample of the CSV
- Were role mappings changed recently

Document 10: Known Limitations and Conservative Answers
================================================================================

If a customer asks whether AtlasOps supports a custom SCIM bridge for regional failover:
- The correct answer is no, that capability is not documented or supported
- AtlasOps supports one standard SCIM endpoint per workspace on Enterprise

If a customer asks whether audit export retention can be overridden on Pro:
- The correct answer is no
- Longer retention requires Enterprise plus the Compliance Archive add-on

If a customer asks whether Pro supports SSO:
- The correct answer is yes
- The configuration path is Settings > Security > Single Sign-On

If a customer asks where API rate limit details are documented:
- The correct answer is that Pro is limited to 120 requests per minute per workspace
- Enterprise is limited to 300 requests per minute per workspace
- Customers should use exponential backoff after HTTP 429

If a customer requests a refund because they were charged twice:
- The correct support behavior is to acknowledge the issue, collect billing identifiers, and escalate
- Support must not promise a refund outcome before finance review
