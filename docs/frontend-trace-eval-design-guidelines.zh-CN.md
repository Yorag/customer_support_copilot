# Trace 与评估页面设计准则

## 1. 目的

本文档把当前 `Trace 与评估` 页面已经验证有效的设计原则沉淀为后续页面改进的参考规范。

这不是一份只针对单页的临时说明，而是控制台后续页面收口时应优先参考的一套界面判断标准。

目标是让其他页面在继续改造时，延续以下方向：

1. 面向真实运行态，而不是演示态。
2. 优先展示可判断、可回溯、可执行的信息。
3. 减少注释性文案、重复文案和伪信息密度。
4. 让页面结构围绕“对象本身”组织，而不是围绕技术事件类别组织。

---

## 2. 总体设计方向

`Trace 与评估` 页当前应被视为控制台页面的设计基准页之一。

它所代表的方向不是“信息越多越好”，而是：

1. 页面必须围绕一个明确对象展开。
2. 主内容区负责解释当前对象发生了什么。
3. 侧边区负责承接选择器、运行摘要、结果评估这类辅助上下文。
4. 一屏内优先给出判断路径，不堆砌术语说明。

对后续页面的启发是：

1. Ticket Detail 应围绕单个 Ticket 的当前状态与动作组织。
2. Gmail Ops 应围绕单次操作或当前运行态组织。
3. Test Lab 应围绕一次测试提交及其结果链路组织。
4. System Status 应围绕当前系统快照与异常信号组织。

---

## 3. 信息组织原则

### 3.1 先按业务对象组织，不先按技术类别组织

`Trace 与评估` 页最重要的收敛经验是：

不要先把信息按 `decision / tool / model / worker` 这种技术分类平铺给用户，再要求用户自己在脑中重新拼出执行过程。

更好的方式是：

1. 先按真实节点或真实对象组织。
2. 再在节点内部展示该对象关联的技术子事件。

后续页面沿用时，应优先判断：

1. 用户当前看的到底是 Ticket、Run、Node、Draft、Scan 还是 System Snapshot。
2. 页面主结构应该直接对应这个对象。
3. 技术细节只能作为二级信息挂在对象之下。

### 3.2 主层只保留判断必须的信息

主层信息应回答：

1. 现在看的是什么。
2. 它当前状态如何。
3. 它经历了哪些关键步骤。
4. 是否存在异常。
5. 下一步该看哪里或操作什么。

不应在主层堆积：

1. 解释性段落。
2. 重复摘要。
3. 同义字段。
4. 对当前对象没有直接帮助的统计。

### 3.3 详情层再展开技术细节

二级详情适合承接：

1. metadata 字段。
2. request id / input ref / output ref。
3. tool / llm / decision 明细。
4. 原始记录。

这类信息默认不应抢占主层视觉焦点。

---

## 4. 展示密度原则

### 4.1 不能靠放大卡片制造“高级感”

这次页面收敛已经证明：

1. 卡片太大但信息太少，会显得空、假、拖沓。
2. 正确做法是控制卡片间距和布局，而不是盲目放大卡片。

后续页面应遵循：

1. 优先调布局密度。
2. 优先让能同行显示的信息同行显示。
3. 对短字段优先采用多列摘要卡。
4. 对长字段采用单独一行或单独区块，不要挤入短字段卡。

### 4.2 留白要服务于层级，不是平均分配

合理留白用于：

1. 区分主次层级。
2. 区分对象卡与详情卡。
3. 区分可操作区域与只读区域。

不合理留白的典型问题：

1. 单卡内容只有一两行，但占据过大垂直空间。
2. 同级卡片全部纵向堆叠，破坏扫描效率。
3. 字段很短，却被拉成整行。

---

## 5. 文案原则

### 5.1 少解释，多命名

页面文案应优先：

1. 准确命名当前对象。
2. 准确命名状态。
3. 准确命名动作。

尽量避免：

1. “这里用于展示……”
2. “你可以在此查看……”
3. “当前运行拆解为 xx 个节点……”
4. 大量陪衬性质的说明文字。

### 5.2 中文用于副标题与辅助说明，主标题可保留英文对象名

对于 trace、node、tool、event 这种带技术辨识度的对象：

1. 主标题可以保留英文原值。
2. 副标题、标签、辅助说明可以中文化。

这样做的原因是：

1. 英文对象名与后端实体一致，方便回查。
2. 中文辅助说明降低阅读门槛。
3. 避免主标题中英混乱或翻译失真。

### 5.3 Tooltip 优于常驻说明文案

对公式、评分口径、扩展解释这类内容：

1. 默认不应常驻占位。
2. 优先用 tooltip / popover 承接。

适用内容包括：

1. 评分公式。
2. 子项解释。
3. 统计口径。
4. 字段来源说明。

---

## 6. 轨迹页面专属准则

### 6.1 左侧轨迹必须围绕节点，而不是围绕事件类型

当前 `Trace 与评估` 页已经确认，轨迹区应遵循：

1. 一张主卡代表一个真实节点。
2. 节点卡只展示节点级摘要。
3. 节点展开后再展示它挂接的 `decision / llm_call / tool_call / worker` 子事件。

不应再回到以下做法：

1. 先把 decision、tool、model、worker 当作一级分组。
2. 让用户自己从事件堆里推断节点执行过程。

### 6.2 节点观察区应先给节点结论，再给事件明细

节点展开区优先顺序应为：

1. 节点名。
2. 节点状态。
3. 节点耗时。
4. 节点 metadata。
5. 关联子事件。
6. 原始记录。

### 6.3 “关键线索”这类重复摘要默认不要保留

若一个摘要块只是把下面已有字段再重写一遍，则应删除。

保留条件只有一个：

1. 它必须比下方字段更快提供新的判断价值。

否则一律视为冗余。

### 6.4 原始记录不是大 JSON 的借口

原始记录存在的意义是：

1. 作为回查依据。
2. 承接开发和诊断需求。

但原始记录不应替代结构化展示。

也就是说：

1. 用户应先看到结构化字段。
2. 再决定是否展开原始记录。

---

## 7. 右侧面板准则

### 7.1 运行浏览器只保留运行选择和最小摘要

右侧 `运行浏览器` 应只承担：

1. Ticket / Run 选择。
2. 当前运行最小摘要。
3. Trace 引用。

不应在这里重复展示：

1. 已在左侧主轨迹中出现的过程信息。
2. 已在评估区出现的评分解释。
3. 与当前选择无关的长说明。

### 7.2 运行评估只展示结果，不展示大段注释

评估区适合展示：

1. 延迟。
2. 资源。
3. 回复质量评分。
4. 轨迹符合度评分。

扩展说明应进入 tooltip，而不是挤占卡片空间。

---

## 8. 冗余判断规则

后续页面改造时，可以用下面这组问题快速判断一个信息块是否该删：

1. 这块内容是否提供了新的判断价值？
2. 这块内容是否只是重复另一个区域已经展示的信息？
3. 这块内容是否只是把字段名写成一句自然语言？
4. 这块内容是否只适合在 hover 或展开态出现？
5. 删除后是否会影响用户完成关键判断？

如果答案接近：

1. “没有新增价值”
2. “只是重复”
3. “只是说明”

则默认应删除或降级。

---

## 9. 后续页面复用建议

后续页面改造时，建议默认复用以下模板判断：

### 9.1 对象页模板

适用于：

1. Ticket Detail
2. Trace
3. Draft Detail
4. Scan Result

结构：

1. 左侧主对象过程或内容。
2. 右侧选择器、摘要、结果。
3. 展开区承接技术细节。

### 9.2 运行态页模板

适用于：

1. Gmail Ops
2. System Status

结构：

1. 顶部当前态摘要。
2. 中部主要运行对象或异常对象。
3. 底部次级统计与日志入口。

### 9.3 评估页模板

适用于所有带评分或指标的页面：

1. 主卡显示结果。
2. Tooltip 显示口径。
3. 原因摘要控制在短句内。
4. 原始数据仅在展开区查看。

---

## 10. 实施约束

后续任何控制台页面改造，若涉及信息重组、摘要卡、详情抽屉、节点观察、评分展示，应优先参考本文，而不是重新发明一套页面逻辑。

如果后续页面要偏离本文规则，应至少满足以下条件之一：

1. 页面对象类型完全不同。
2. 本文规则在该页面会显著损害可读性。
3. 偏离后能给出更高的信息效率和更低的冗余。

否则默认应保持与 `Trace 与评估` 页一致的设计判断。
