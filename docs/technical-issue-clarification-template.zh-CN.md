# 技术故障补充信息澄清模板

## 1. 使用场景

当邮件已被识别为 `technical_issue`，但系统判断当前诊断信息不足以形成可信答复时，使用本模板向用户请求补充信息。

触发原则：

1. 不改变主路由，仍保持 `primary_route = technical_issue`。
2. 只在信息不足时触发，不应对所有技术故障一律发起澄清。

---

## 2. 最小补充信息清单

建议至少请求以下信息中的相关项：

1. 问题发生时的具体操作步骤。
2. 实际出现的报错信息或异常现象。
3. 期望结果是什么。
4. 实际结果是什么。
5. 问题发生的时间范围。
6. 相关账号、租户、项目或环境信息。
7. 报错截图或录屏。

---

## 3. 模板示例

```text
您好，

我们已经收到您的问题。为了更准确地定位原因并尽快协助您处理，请您补充以下信息中与该问题相关的内容：

1. 您当时的操作步骤；
2. 具体的报错提示或异常现象；
3. 您预期应出现的结果，以及实际发生的结果；
4. 问题出现的大致时间；
5. 相关账号、租户、项目或运行环境信息；
6. 如方便，请附上报错截图或录屏。

收到这些信息后，我们会继续帮您排查。
```

---

## 4. 生成要求

1. 语气应简洁、礼貌、可执行。
2. 不要一次性索取与当前问题无关的大量信息。
3. 如果已知问题类型较明确，应优先索取最关键的 2 到 4 项，而不是机械罗列全部字段。
4. 不要在信息不足时假设结论或给出未经证实的修复方案。
